# 作者复活引擎设计方案

## 1. 目标定义

产品目标：

- 用户上传一部长篇断更小说。
- 系统读完已有正文。
- 自动生成像原作者复活一样的下一章。
- 用户可以控制人物走势与情节方向。
- 生成结果能通过盲看 1000 字测试。

核心验收标准：

- 盲看 1000 字，熟读者不容易分辨 AI 续写。
- 不出现 AI 套话、解释、总结、现代分析腔。
- 人物声口稳定，角色不会突然说不像自己的话。
- 长篇设定不乱，死人不复活，信息不穿帮，道具不重复。
- 用户能导演走势，但导演意图不会以现代概念词露进正文。
- 成本可控，300 万字小说也能跑。

## 2. 总体架构

不要再用重型 LightRAG 图谱当主线。新架构应该是混合式：

```text
原文输入
  ↓
文本清洗与章节切分
  ↓
长上下文读书笔记
  ↓
薄知识图谱 + 时间线 + 人物状态卡 + 伏笔队列
  ↓
作者声纹编译器
  ↓
章节导演器
  ↓
局部原文检索
  ↓
GPT-5.5 生成正文
  ↓
机械风格闸门
  ↓
盲测判别器
  ↓
失败重写循环
  ↓
最终章节
```

一句话：

```text
状态系统保证不乱；
声纹系统保证像；
盲测系统保证不是自嗨。
```

## 3. 模块设计

### 3.1 文本预处理层

职责：

- 统一繁简策略。
- 统一标点策略。
- 自动识别章节标题。
- 去掉广告、作者有话说、网页水印。
- 保留正文原貌。
- 建立 `chapter_id`、`scene_id`、`paragraph_id`。

输出示例：

```json
{
  "work_id": "hongloumeng",
  "chapters": [
    {
      "chapter": 80,
      "title": "美香菱屈受贪夫棒 王道士胡诌妒妇方",
      "start_char": 612000,
      "end_char": 620466,
      "paragraphs": []
    }
  ]
}
```

### 3.2 薄知识图谱

只保硬事实，不抽软象征。

保留：

- 人物。
- 身份。
- 关系。
- 生死状态。
- 阵营。
- 地点。
- 道具。
- 能力体系。
- 已知信息。
- 禁止违背事实。

不要保留：

- “宝玉与芙蓉花存在悲悼象征关系”。
- “大观园代表女性命运共同体”。
- “此处体现封建压迫”。

这些软解释会污染正文。

示例：

```json
{
  "characters": {
    "林黛玉": {
      "status": "alive",
      "location": "潇湘馆",
      "role": "贾府寄居表姑娘",
      "knows": ["宝玉改芙蓉诔句后心中狐疑"],
      "must_not": ["直白表白", "突然主动争婚"]
    }
  }
}
```

### 3.3 时间线 Ledger

长篇不乱，关键靠时间线。

```json
{
  "chapter": 80,
  "events": [
    {
      "event": "香菱被夏金桂折磨，改名秋菱",
      "characters": ["香菱", "夏金桂", "薛蟠"],
      "state_changes": {
        "香菱": "处境恶化，身体与精神受损"
      },
      "open_threads": ["香菱薄命线", "薛家内宅冲突"]
    }
  ]
}
```

写新章前只取：

- 最近 10-20 章事件。
- 当前出场人物相关事件。
- 相关伏笔事件。

### 3.4 人物状态卡

人物状态卡比图谱更重要。每个主要人物一张卡。

示例：

```markdown
# 林黛玉

当前处境：
- 病弱，寄居贾府。
- 与宝玉情感深，但受礼法与婚姻不自主约束。

已知信息：
- 知道宝玉改“茜纱窗下，我本无缘”句后心中不安。
- 对宝玉心意敏感，但不会正面索问。

声口：
- 尖新、敏感、半真半讥。
- 常以玩笑、反话、诗句遮掩真心。
- 说话不做现代心理分析。

不可写：
- 不直白说“我没有安全感”。
- 不直接总结“我们被封建礼教压迫”。
```

### 3.5 伏笔队列

长篇续写最容易忘伏笔，必须单独管理。

```json
{
  "open_threads": [
    {
      "id": "T_furong_elegy",
      "name": "芙蓉女儿诔",
      "introduced_at": 78,
      "last_advanced": 80,
      "risk": "宝玉将晴雯、黛玉、群芳薄命混为一体",
      "allowed_moves": [
        "用旧物触发",
        "用诗句误会触发",
        "用黛玉反应触发"
      ],
      "forbidden_moves": [
        "直接解释象征意义",
        "让人物现代化总结命运"
      ]
    }
  ]
}
```

每章只推进 1-3 条伏笔，不要全塞。

### 3.6 作者声纹编译器

这是新系统的核心。它不问“发生了什么”，而问“怎么写”。

输出：

- 叙述人称与视角。
- 句长分布。
- 虚词密度。
- 对白比例。
- 常用转折。
- 常用章回推进方式。
- 诗词/判词插入方式。
- 每个人物声口。
- 禁止现代词。
- 常见段落结构。

示例声纹规则：

```json
{
  "narration_patterns": [
    "且说……",
    "谁知……",
    "原来……",
    "一面……一面……",
    "不觉……",
    "说着……"
  ],
  "modern_forbidden": [
    "心理压力",
    "关系变化",
    "信息反转",
    "情绪价值",
    "创伤",
    "压迫感",
    "安全感"
  ],
  "style_metrics": {
    "avg_sentence_length": 28,
    "dialogue_ratio": 0.42,
    "de_density_max": 0.007
  }
}
```

### 3.7 长上下文读书笔记

用便宜 1M 上下文模型，不用 GPT-5.5。

任务：

- 通读全文。
- 产出人物状态。
- 产出伏笔队列。
- 产出世界规则。
- 产出章节节奏。
- 产出声口初稿。

GPT-5.5 不该做这个阶段。它太贵，而且收益不匹配。

### 3.8 局部原文检索

生成第 N+1 章前，系统必须取原文证据。

比如写《红楼梦》第 81 回，要取：

- 第 74 回：抄检大观园。
- 第 77 回：晴雯被逐。
- 第 78 回：芙蓉女儿诔。
- 第 79-80 回：香菱、夏金桂。
- 宝黛典型对白片段。
- 迎春相关片段。

检索不是为了“知识”，而是为了让 GPT-5.5 贴着原文文气写。

### 3.9 隐藏导演层

用户可能说：

```text
我要宝玉精神崩溃，黛玉误会加深，迎春线更惨。
```

不能原样塞进正文 prompt。要翻译成古典场面约束：

- 宝玉不说“我崩溃了”，而是因旧物、残花、病后昏沉触发。
- 黛玉不说“我误会了”，而是用冷笑、反问、避开话头表现。
- 迎春不控诉孙家，只藏伤、低头、替别人遮掩。

现代导演语言必须在进入正文前被“作品内部化”。

### 3.10 正文生成器

GPT-5.5 只在这里重用。

输入：

- 当前章节目标。
- 人物状态卡。
- 伏笔队列。
- 局部原文片段。
- 声纹手册。
- 禁止词。
- 导演约束。

输出只允许正文。

硬要求：

- 不输出标题。
- 不输出解释。
- 不输出总结。
- 不输出“以下为”。
- 不使用现代分析词。
- 不直接说主题。
- 不写剧情梗概。
- 不背离人物声口。

### 3.11 机械风格闸门

先用程序拦低级错误：

- 是否有“以下为”“本文”“总结”等包装语。
- 是否有现代抽象词。
- `的/了/着/这/是` 密度是否偏离原文。
- 平均句长是否偏离。
- 对白称谓是否错。
- 繁简是否混乱。
- 是否连续出现解释性段落。

机械闸门只能拦明显错误，不能判断文气。

落地要求：

- 风格闸门失败不能被普通剧情质量分覆盖成通过。
- 短章修订必须带源文本章节长度目标，不能靠总结或开篇回顾凑字。
- 闸门报告要能进入会话状态，让批量生成时的失败和告警可见。
- 连续章节还要拦近章开头重复；长篇续写不能连续几章用同一套“话说……”转场糊过去。
- 具体实现状态和验证记录见 `docs/revival-validation-status.md`。

### 3.12 盲测判别器

这是最终质量核心。

流程：

```text
1. 从生成正文截取 1000 字。
2. 随机抽 3 段原文 1000 字。
3. 打乱顺序。
4. 让 GPT-5.5 判断哪段不是原作者。
5. 如果判中，要求指出不像的句子。
6. 把失败原因转成重写约束。
7. 重写，再盲测。
```

判别提示不能问：

```text
这段写得好不好？
```

要问：

```text
四段中哪一段最不像同一作者？依据只看文体、句法、声口、叙述节奏，不看剧情。
```

通过条件：

- 判别器无法稳定识别生成段。
- 或多轮判别置信度低于阈值。

## 4. 300 万字小说处理策略

300 万字不能每次全读。采用分层记忆：

```text
全书层：世界规则、主线、人物总表
卷层：阶段冲突、阵营变化
最近层：最近 20 章详细事件
人物层：每个重要人物状态卡
伏笔层：所有未收线索
原文层：可检索片段库
```

生成一章时只取：

- 全书硬规则。
- 当前卷摘要。
- 最近 20 章。
- 出场人物卡。
- 相关伏笔。
- 相关原文片段。

这样成本稳定，不会随全文线性爆炸。

## 5. 模型分工

推荐分工：

```text
便宜长上下文模型：
- 通读
- 读书笔记
- 初步状态抽取
- 初步人物卡
- 初步伏笔表

本地程序：
- 章节切分
- 文体统计
- 机械闸门
- BM25/向量检索
- 状态版本管理

便宜模型：
- 一致性检查
- 硬事实核对
- 伏笔更新建议

GPT-5.5：
- 正文生成
- 声纹判别
- 重写决策
```

成本原则：

```text
便宜模型负责读和整理；
GPT-5.5 负责审美和最终表达。
```

## 7. 核心流程

生成下一章：

```text
1. 用户指定方向。
2. 系统读取当前状态。
3. 导演层把用户方向翻译成场面约束。
4. 检索相关原文。
5. GPT-5.5 生成初稿。
6. 机械闸门检查。
7. 盲测判别。
8. 不通过则生成重写指令。
9. 最多循环 3-5 次。
10. 通过后写入最终章节。
11. 更新 timeline、人物卡、伏笔队列、薄 KG。
```

## 8. 为什么这比当前方案好

当前问题：

```text
LightRAG 把作品抽成实体关系；
生成器按剧情目标写；
质检器奖励剧情完整；
结果像剧情续写，不像原作者。
```

新方案：

```text
薄 KG 防乱；
时间线防穿帮；
声纹手册防不像；
局部原文防漂；
盲测防自嗨；
GPT-5.5 只花在关键审美环节。
```

## 9. 实施计划

### 第一阶段：红楼专用验证版

- 暂停无质量门兜底的后 40 回批量生成。
- 标记 81-84 为失败样本，并把第 120 回作为末段文风漂移与短章回归样本。
- 建红楼 `style_bible`。
- 建人物声口卡：宝玉、黛玉、宝钗、袭人、王夫人、凤姐、迎春、香菱。
- 建 `forbidden_words`。
- 做 1000 字盲测器。
- 只重写第 81 回。
- 增加末段回归样本，验证长链路写到后段时是否仍能守住源文风和章节尺度。

### 第二阶段：通用断更小说版

- 自动章节切分。
- 自动人物卡。
- 自动伏笔队列。
- 自动状态更新。
- 用户导演输入。
- 多轮盲测重写。

### 第三阶段：长篇规模化

- 支持 300 万字。
- 支持卷层记忆。
- 支持最近 20 章窗口。
- 支持人物出场检索。
- 支持成本预估。

## 10. 产品判断

这个产品真正的护城河不是“能续写”。

市面上很多模型都能续写。

真正难的是：

```text
读者看 1000 字，觉得作者真的回来了。
```

所以系统指标必须从：

```text
剧情完整度
```

升级成：

```text
作者相似度 + 盲测不可识别率
```

否则继续堆 RAG、堆图谱、堆大模型，都会跑偏。
